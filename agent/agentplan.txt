Dataset: what you're actually working with
Your sqllab_agent.csv is a Nigerian banking transactions dataset — 1000 rows with fields: tx_id, timestamp, bank_code, bank_name, type (DEPOSIT/WITHDRAWAL), channel_type (AGENT/COUNTER), location_type, state, amount, currency, status, processing_fee, latitude, longitude, and Kafka metadata. This dataset is already loaded into PostgreSQL inside Docker under the superset_light database and registered as a Superset dataset. This is important — it means the Dataset Agent doesn't need to create anything, only discover and read what already exists.

The correct pipeline logic, phase by phase
Phase 0 — Startup health check (run once on boot)
Before anything else, your pipeline pings two things. First, it calls the MCP health_check tool at localhost:5008/mcp — if this fails, the whole pipeline stops immediately and surfaces the error clearly. Second, it calls localhost:9001/api/v1/security/login with admin/admin to get a JWT token — this token is used for any REST API fallbacks you need (described below). This phase runs once, not per query.
Phase 1 — Orchestrator agent: intent parsing and plan generation
The user sends a natural language request like "show me DEPOSIT vs WITHDRAWAL volume by state, and a time series of transaction count by day". The Orchestrator is a single LLM call to your local gpt-20b model. Its system prompt is tightly constrained: it must respond only in JSON, no preamble, no markdown fences. The JSON plan specifies: which dataset names to find, what charts to create (each with name, chart_type, metric, dimension, time_column if temporal), and the dashboard title.
The system prompt must also communicate the dataset's known domain — banking transactions, Nigerian states, NGN amounts — so the model produces realistic column names rather than hallucinating generic ones like category or revenue.
Phase 2 — Dataset agent: schema discovery via MCP
The Dataset Agent calls list_datasets via MCP and matches the plan's dataset names against what Superset actually has. For your setup, it should find the sqllab_agent table registered under the superset_light database.
Then it calls get_dataset_info with the matched dataset's ID. This returns the exact column names as Superset knows them — crucial because generate_chart will reject anything that doesn't match precisely. The agent builds a schema map: { column_name → type } for all columns. This schema map is passed to every downstream agent.
Critical correctness note: your dataset has a timestamp column which Superset may or may not have parsed as a datetime type depending on how the CSV was ingested. The Dataset Agent must check the type of timestamp from get_dataset_info and flag it if it's stored as VARCHAR rather than TIMESTAMP — because temporal charts will silently fail with the wrong type. If it's stored as string, the SQL Agent needs to handle casting.
Phase 3 — SQL agent: column validation (not optional)
The SQL Agent runs a lightweight probe query for each chart spec using execute_sql over MCP. It constructs a minimal SELECT {dimension}, {metric_expression} FROM sqllab_agent GROUP BY {dimension} LIMIT 5. This achieves three things at once: (1) confirms the column names exist exactly as named, (2) confirms the aggregation works (SUM of amount, COUNT of tx_id, etc.), (3) returns a preview of real values the Orchestrator can use to sanity-check the plan.
For the execute_sql call, you need the database_id — this comes from the dataset info returned in Phase 2, specifically dataset_info["result"]["database"]["id"]. Do not hardcode this.
The SQL Agent should also detect if the query returns zero rows — which would mean the filter or dimension is wrong — and feed this back to the Orchestrator before wasting a generate_chart call.
Phase 4 — Chart agent: creation with self-correction
This is the most failure-prone step. The Chart Agent calls generate_chart for each chart in the plan. The form_data params must match the viz type exactly.
For your dataset, the useful viz types and their correct Superset identifiers are:

Bar chart comparing values across states → echarts_timeseries_bar or dist_bar
Time series of transaction count by day → echarts_timeseries_line
Pie/donut of transaction type split → pie
Table of top banks by volume → table

The metrics field must use Superset's metric format — either a simple string like "count" if Superset has a pre-defined metric, or an adhoc metric object specifying expressionType: "SIMPLE", column, and aggregate. The SQL Agent's probe in Phase 3 gives you the right column names to use here.
Self-correction loop: When generate_chart returns an error (e.g. "column not found, did you mean amount?"), the Chart Agent must not crash. It extracts the suggestion from the structured error, updates the params, and retries — up to 3 times. After 3 failures it marks that chart as failed and continues with the others. The Orchestrator gets a final report showing which charts succeeded and which failed, so the user knows what they got.
Phase 5 — Dashboard agent: assembly
Once all chart IDs are collected, the Dashboard Agent calls generate_dashboard with the list of IDs and published: True. It returns the dashboard ID, and the final URL is http://localhost:9001/superset/dashboard/{id}/.
The Dashboard Agent should call get_dashboard_info afterward to confirm the dashboard actually exists in Superset before returning the URL to the user. This catches silent failures.
Phase 6 — Orchestrator: result reporting
The Orchestrator assembles the final output: the dashboard URL, a list of successfully created charts, any charts that failed with their reasons, and the SQL previews from Phase 3 so the user can verify the data looked right. This is not just a URL — it's a structured summary the user can act on.

The self-correction loop (most important design decision)
Every agent in this pipeline must return a structured result object, not just a value or an exception. The result has three fields: success (bool), data (the payload if successful), and error (structured error from MCP if not). The Orchestrator inspects these results and decides whether to retry with adjusted parameters, skip and continue, or abort and report.
This loop is what makes the system agentic. The Orchestrator can re-prompt the LLM with the error context — e.g. "the column state was not found, the available columns are: [tx_id, bank_name, type, channel_type, location_code, state, amount], please revise the plan" — and get a corrected plan without requiring user intervention.
